## CRITICAL — 编译错误

### 问题一：`PageResult.of()` 调用参数数量不匹配（3处）

**根本原因：** `PageResult.of(IPage<T> page)` 是单参数方法，但以下 3 处调用传入了 2 个参数：

| 文件 | 行号 | 错误调用 |
|------|------|---------|
| `dafuweng-finance/.../CommissionServiceImpl.java` | 43 | `PageResult.of(result, commissionConverter.toVOList(result.getRecords()))` |
| `dafuweng-finance/.../LoanAuditServiceImpl.java` | 194 | `PageResult.of(result, loanAuditConverter.toSimpleVOList(result.getRecords()))` |
| `dafuweng-finance/.../BankServiceImpl.java` | 56 | `PageResult.of(result, bankConverter.toVOList(result.getRecords()))` |

**`PageResult` 当前 API：**
```java
public static <T> PageResult<T> of(IPage<T> page)  // 单参数
public static <T> PageResult<T> empty(long size)
```

`PageResult.of()` 只接受 `IPage<T>` 参数，接收后内部将 `IPage` 的 `records`/`total`/`size`/`current` 拆解填充到 `PageResult`。传入 converter 后的 VO List 与 `PageResult.of()` 的签名完全不匹配。

**修复方法：**

需先调用 converter，再用 4 参数构造方法构建 `PageResult`：

```java
// 错误（编译失败）：
return PageResult.of(result, commissionConverter.toVOList(result.getRecords()));

// 正确：
List<CommissionRecordVO> voList = commissionConverter.toVOList(result.getRecords());
return new PageResult<>(voList, result.getTotal(), result.getSize(), result.getCurrent());
```

## HIGH — 逻辑错误

### 问题三：`LoanAuditController` 分页参数构造方式不规范

**文件：** `dafuweng-finance/.../controller/LoanAuditController.java:21-26`

```java
public R<PageResult<LoanAuditSimpleVO>> received(
        @RequestParam(defaultValue = "1") Integer pageNum,
        @RequestParam(defaultValue = "10") Integer pageSize) {
    PageResult<LoanAuditSimpleVO> page = new PageResult<>();
    page.setCurrent(pageNum);
    page.setSize(pageSize);
    return R.ok(loanAuditService.received(page));
}
```

问题：`new PageResult<>()`（无参构造）创建的对象只有 `current` 和 `size` 被设置，`total`、`records`、`pages`、`hasNext`、`hasPrevious` 均为默认值 0/null。这不是错误（`LoanAuditServiceImpl:189` 实际使用的是 `page.getCurrent()` 和 `page.getSize()`），但是：
1. 创建了多余的无用对象
2. 如果 `LoanAuditServiceImpl.received()` 的实现发生变化（例如直接使用 `page.getTotal()`），会返回错误数据

**修复方法：** 直接传 `pageNum` 和 `pageSize` 给 service，或者 service 直接接收两个 Integer 参数：

```java
// 方式一：修改 controller
public R<PageResult<LoanAuditSimpleVO>> received(
        @RequestParam(defaultValue = "1") Integer pageNum,
        @RequestParam(defaultValue = "10") Integer pageSize) {
    return R.ok(loanAuditService.received(pageNum, pageSize));
}

// 方式二：保留现有调用方式（可接受，但建议改为方式一）
```

### 问题四：多处 `page()` 方法返回类型与 entity 不对齐

**问题概述：** 以下 `page()` 方法返回 `PageResult<VO>`，但内部使用 `PageResult.of(IPage<Entity>)`：

| 文件 | 方法 | 返回类型 | 内部实际 |
|------|------|---------|---------|
| `CustomerServiceImpl.page()` | `PageResult<CustomerVO>` | `PageResult.of(IPage<Customer>)` |
| `ContractServiceImpl.pageContract()` | `PageResult<ContractVO>` | `PageResult.of(IPage<Contract>)` |

**`PageResult.of(IPage<Customer>)` 返回 `PageResult<Customer>`，不是 `PageResult<CustomerVO>`。**

这会导致**编译类型不匹配**（`PageResult<Customer>` 无法赋值给 `PageResult<CustomerVO>`）。如果当前能编译通过，说明存在隐式转换或者 `CustomerVO` 和 `Customer` 之间有继承关系。

**验证方法：**
```bash
# 检查 CustomerVO 是否继承 Customer
grep "class CustomerVO" dafuweng-sales/src/main/java/com/dafuweng/sales/domain/vo/CustomerVO.java
```

如果 `CustomerVO extends Customer`，则不存在类型问题；如果没有继承关系，则编译失败。

**修复方法（如果确实存在类型不匹配）：**
```java
// 错误：
IPage<Customer> pageResult = customerMapper.selectPage(page, wrapper);
return PageResult.of(pageResult);  // 返回 PageResult<Customer>

// 正确：
IPage<Customer> pageResult = customerMapper.selectPage(page, wrapper);
List<CustomerVO> voList = customerConverter.toVOList(pageResult.getRecords());
return new PageResult<>(voList, pageResult.getTotal(), pageResult.getSize(), pageResult.getCurrent());
```

## 中等优先级

### 问题五：多处 MapStruct Converter toVO() 方法字段不完整

以下 VO 的部分字段在 entity 中不存在（来自 join 查询或业务计算），MapStruct 无法自动映射，这些字段运行时为 null：

| VO | 缺失字段（部分） | 影响 |
|----|----------------|------|
| `CustomerVO` | `customerTypeDesc`、`salesRepName`、`deptName`、`zoneName`、`intentionLevelDesc`、`statusDesc`、`nextFollowUpDate`、`publicSeaReason`、`annotations`、`source`、`loanIntentionAmount`、`loanIntentionProduct`、`createdByName` | 列表接口返回的 VO 数据不完整 |
| `ContactRecordVO` | `customerName`、`salesRepName`、`contactTypeDesc`、`intentionBeforeDesc`、`intentionAfterDesc`、`followUpDate`、`attachmentUrls` | 洽谈记录详情数据缺失 |
| `PerformanceVO` | `contractNo`、`salesRepName`、`deptName`、`zoneName`、`statusDesc`、`calculateTime`、`confirmTime`、`grantTime` | 绩效数据不完整 |
| `AccountVO` | `deptName`、`zoneName`、`statusDesc`、`roles`、`lastLoginTime`（类型差异） | 账户信息展示缺失 |
| `BankVO` | `statusDesc` | 银行状态描述缺失 |
| `DepartmentVO` | `parentName`、`zoneName`、`managerName`、`statusDesc`、`children` | 部门信息不完整 |

**原因分析：** 这些字段通常需要多表 join 查询（如 `customer` join `sys_user` 获取 `salesRepName`），或者需要枚举转换（如 `customerType` → `customerTypeDesc`），MapStruct 只做同名字段映射，无法处理 join 和枚举翻译。

**修复方向：** 有两种方案
1. **Service 层手动填充：** 在 service 层查询后手动 set 这些字段（适合少量字段）
2. **SQL join 查询：** 修改 mapper XML，使用 resultMap 做 join 查询（适合大量字段）

具体采用哪种方案，需要根据业务场景决定。
