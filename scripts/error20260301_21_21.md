# 依赖问题处理方案（第二轮）

## 问题描述

上一次对 `dafuweng-common` 的重构（合并 common-core/common-mybatis/common-redis/common-feign 为单一模块）完成之后，出现了以下问题：

1. **微服务无法依赖 common 包** — `dafuweng-common` 无法被 `dafuweng-sales`、`dafuweng-finance`、`dafuweng-system` 等模块正常依赖。
2. **POM 版本定义问题** — `dafuweng-common/pom.xml` 中 parent 版本引用使用了不规范的写法。

---

## 问题一：`dafuweng-common/pom.xml` 的 parent 版本引用

### 当前（错误写法）

```xml
<parent>
    <groupId>com.dafuweng</groupId>
    <artifactId>dafuweng-parent</artifactId>
    <version>${project.version}</version>   <!-- 危险：引用自身版本 -->
    <relativePath>../pom.xml</relativePath>
</parent>
```

`dafuweng-common` 自身没有 `<version>` 元素（通过 parent 继承），在 parent 引用的 `<version>` 中使用 `${project.version}` 是不规范的。虽然在 Maven 继承链中这会解析为从 parent 继承的版本（`1.0.0`），但这种写法会导致：
- Maven 生命周期中 parent 版本解析的歧义
- IDE（如 IntelliJ）有时无法正确处理这种写法
- 读者难以理解模块的实际版本来源

### 正确写法

```xml
<parent>
    <groupId>com.dafuweng</groupId>
    <artifactId>dafuweng-parent</artifactId>
    <version>1.0.0</version>
    <relativePath>../pom.xml</relativePath>
</parent>
```

## 问题二：`hutool-all` 和 `fastjson2` 版本未使用 parent 属性

### 当前（不一致）

`dafuweng-common/pom.xml`:
```xml
<dependency>
    <groupId>cn.hutool</groupId>
    <artifactId>hutool-all</artifactId>
    <version>5.8.26</version>   <!-- 硬编码，与 parent.properties 不一致 -->
</dependency>
<dependency>
    <groupId>com.alibaba.fastjson2</groupId>
    <artifactId>fastjson2</artifactId>
    <version>2.0.47</version>  <!-- 硬编码，与 parent.properties 不一致 -->
</dependency>
```

`dafuweng-parent/pom.xml` 中已经定义了：
```xml
<fastjson2.version>2.0.47</fastjson2.version>
<hutool.version>5.8.26</hutool.version>
```

### 正确写法

```xml
<dependency>
    <groupId>cn.hutool</groupId>
    <artifactId>hutool-all</artifactId>
    <version>${hutool.version}</version>
</dependency>
<dependency>
    <groupId>com.alibaba.fastjson2</groupId>
    <artifactId>fastjson2</artifactId>
    <version>${fastjson2.version}</version>
</dependency>
```

## 问题三：多模块构建顺序导致的依赖不可用

### 根本原因

在 Maven 多模块项目中，如果直接对子模块执行 `mvn compile`（而不从 parent 开始构建），Maven 无法找到依赖模块的 artifact，因为该 artifact 尚未安装到本地仓库。

例如：直接对 `dafuweng-sales` 执行 `mvn compile`，此时 `dafuweng-common` 尚未 install，所以 Maven 报错找不到 `com.dafuweng:dafuweng-common:1.0.0`。

### 正确构建方式

**方式一：从项目根目录构建（推荐）**
```bash
cd 项目根目录
mvn clean install -DskipTests
```
Maven 会按照 `dafuweng-parent` 的 `<modules>` 列表顺序构建所有模块，`dafuweng-common` 会先构建并 install 到本地仓库，之后的服务模块可以正常依赖它。

**方式二：先 install common，再构建服务**
```bash
cd 项目根目录
mvn clean install -pl dafuweng-common -DskipTests   # 先构建并 install common
mvn clean install -pl dafuweng-sales -DskipTests   # 再构建 sales
```
